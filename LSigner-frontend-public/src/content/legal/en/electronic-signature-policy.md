## 1. Object

## 2. Legal classification of the firm

In accordance with Regulation (EU) No 910/2014 (eIDAS), there are three levels of electronic signature: Simple (SES), Advanced (AES), and Qualified (QES).

The signature generated by LSigner is classified, as of the date of this document, as a **Simple Electronic Signature (SES)**, in accordance with Article 3.10 of eIDAS, since the signature creation data (cryptographic key) is centrally managed by LSigner and is not yet exclusively and individually linked to each signer thru their own unique key.

However, it is a **reinforced SES**: the process incorporates multiple technical guarantees—multi-factor authentication, OTP verification, detailed forensic evidence, and verifiable cryptographic signature—that far exceed the requirements for a standard simple electronic signature (like a simple click or checkbox), providing a level of traceability and evidential strength comparable, in practice, to more advanced systems, albeit without the reinforced legal presumption granted by eIDAS to AES and QES signatures.

According to Article 25.1 of eIDAS, the legal effects or admissibility as evidence in judicial proceedings of an electronic signature cannot be denied solely because it is simple. Its final probative value will be assessed according to the general rules of electronic evidence, precisely considering the robustness of the evidence described in this document.

LSigner plans to evolve toward a model of individual keys per User with rotation capability, which would allow, in the future, to reclassify the service as Advanced Electronic Signature (AES), provided that the rest of the requirements of Article 26 of eIDAS are also met. Any change in this classification will be communicated to the Users thru the update of this document.

## 3. Description of the signing process

1. **Authentication:** The User logs in using their own credentials and a second authentication factor (MFA). **Authentication:** The User logs in using their own credentials and a second authentication factor (MFA).
2. **Opening the document:** when opening the document to be signed, the system logs an access event (see section 4.1), regardless of whether the User ultimately decides to sign, reject, or abandon the process. **Opening the document:** upon opening the document to be signed, the system records an access event (see section 4.1), regardless of whether the User ultimately decides to sign, reject, or abandon the process.
3. **OTP Verification:** before applying the signature, LSigner generates a one-time code (OTP) and sends it to the User's verified contact channel (email). The User must enter this code to continue. The OTP is generated, managed, and validated entirely by LSigner systems.
4. **Application of the signature:** once the OTP is verified, the LSigner backend applies a cryptographic signature to a canonical representation of the operation (see section 4.3) and generates the signed PDF document.
5. **Evidence Record:** The set of contextual and cryptographic evidence described in section 4 is stored immutably.

## 4. Recorded evidence

### 4.1 Document Access Event (`ACCESS_OPENED`)

Regardless of whether the document is signed, each time the recipient opens the document, a record is generated with:

| Field               | Description                                                                     |
| ------------------- | ------------------------------------------------------------------------------- |
| `ip`                | IP address of the signer at the time of access                                  |
| `user_agent`        | Browser and operating system used                                               |
| `is_first_access`   | Indicates whether it is the first access to the document or a subsequent access |
| `first_accessed_at` | Timestamp of the first access                                                   |
| `last_accessed_at`  | Timestamp of the most recent access                                             |

### 4.2 Contextual Evidence of the Signature

At the moment of signing, rejecting, or revoking, an artifact (`DocumentSignedArtifact`) is generated that includes the following contextual evidence:

| Field                                               | Description                                                                                          |
| --------------------------------------------------- | ---------------------------------------------------------------------------------------------------- |
| `verification_method`                               | Verification method used (`"OTP"`)                                                                   |
| `verification_reference`                            | Masked reference of the channel where the OTP was sent (email or another partially obscured channel) |
| `recipient_email`                                   | Signer's email address                                                                               |
| `reciIP Address from which the action was performed |
| `user_agent`                                        | Browser and operating system used                                                                    |
| `signed_at`                                         | Exact timestamp (ISO 8601 format) of the action                                                      |
| `mime_type`                                         | Document file type                                                                                   |
| `original_filename`                                 | Original file name                                                                                   |
| `reason`                                            | (Only in rejection/revocation) reason voluntarily provided by the User                               |
| `state`                                             | State of the operation: `SIGNED`, `REJECTED`, or `REVOKED`                                           |
